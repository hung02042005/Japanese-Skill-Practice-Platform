# AI Development Rules

## General Principle

Khi phát triển tính năng mới, AI chỉ được thay đổi những file liên quan trực tiếp đến chức năng được yêu cầu.

Mọi thay đổi phải đảm bảo:

* Không làm hỏng chức năng hiện có.
* Không làm thay đổi hành vi của chức năng khác.
* Không thay đổi Database.
* Không refactor ngoài phạm vi yêu cầu.

---

# Database Protection

AI tuyệt đối không được:

* Tạo bảng mới.
* Xóa bảng.
* Sửa cấu trúc bảng.
* Thêm cột.
* Xóa cột.
* Đổi kiểu dữ liệu.
* Tạo migration.
* Sửa migration.
* Chạy migration.
* Thay đổi schema database.

Nếu yêu cầu cần thay đổi database:

STOP.

Thông báo cho người dùng rằng tính năng này yêu cầu thay đổi database và chờ phê duyệt.

---

# Code Modification Policy

AI được phép:

* Sửa file thuộc chức năng đang thực hiện.
* Thêm method mới.
* Thêm class mới.
* Thêm service mới.
* Thêm DTO mới.
* Thêm validation mới.
* Thêm test mới.
* Sửa bug trong phạm vi chức năng được yêu cầu.

AI không được phép:

* Refactor toàn bộ module.
* Đổi tên class không liên quan.
* Đổi tên package không liên quan.
* Xóa code không liên quan.
* Sửa các chức năng khác ngoài phạm vi yêu cầu.

---

# Scope Restriction

Chỉ sửa các file cần thiết để hoàn thành yêu cầu.

Không được:

* Chỉnh sửa hàng loạt nhiều module.
* Thay đổi kiến trúc hệ thống.
* Tối ưu hóa code ngoài phạm vi yêu cầu.
* Sửa lỗi không liên quan đến task hiện tại.

---

# Existing Function Protection

Mọi chức năng hiện có phải tiếp tục hoạt động như trước.

AI phải ưu tiên:

1. Thêm code mới.
2. Mở rộng code hiện có.
3. Chỉ sửa code hiện có khi thực sự cần thiết cho yêu cầu.

---

# Before Making Changes

AI phải tự kiểm tra:

* Có thay đổi Database không?
* Có tạo migration không?
* Có ảnh hưởng chức năng khác không?
* Có refactor ngoài phạm vi task không?

Nếu câu trả lời là YES:

STOP và giải thích lý do trước khi tiếp tục.

---

# Golden Rule

Được phép sửa code trong phạm vi chức năng được yêu cầu.

Không được thay đổi Database.

Không được ảnh hưởng chức năng khác.

Không được refactor ngoài phạm vi task.
